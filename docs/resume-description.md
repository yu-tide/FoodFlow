# FoodFlow — 简历项目描述

## 项目名称

FoodFlow — AI Workflow 驱动的饮食记录与营养分析系统

## 一句话简介

全栈个人项目：通过「图片上传 → OCR → AI 视觉识别 → 营养估算 → AI 总结」的 7 阶段异步流水线，将食物图片自动转化为结构化营养数据，支持 AI 饮食助手对话和每周营养统计分析。

## 简历 Bullet Points（任选 3-5 条）

- **设计并实现了完整的 7 阶段异步分析流水线**：基于 Celery + Redis 的分布式任务队列，每张食物图片经过 OCR 文字提取、AI 视觉识别、多源结果融合、营养数据库匹配和 AI 摘要生成，进度实时可追踪，支持失败重试
- **构建了多层安全架构的 Agentic AI 助手**：包含 Reasoning Gate（意图分类，14 种请求类型）、Planner（结构化计划生成）、Tool Registry（工具注册表，5 个只读工具 + 4 个可写动作）、RAG 知识库检索、Action Audit（操作审计日志 + 前后快照）和 Post-Action Observer（操作后状态重读）
- **实现了个性化食物决策引擎**：综合用户热量预算、当前时间、食物风险标签（高油/高糖/高咖啡因）、近期饮食模式、用户目标类型（减脂/增肌/维持），给出个性化的「能不能吃」判断和替代建议
- **全栈独立开发**：FastAPI + SQLAlchemy(async) + PostgreSQL（11 个数据模型）后端，Next.js 15 + React 19 + TypeScript + TailwindCSS + Recharts 前端，前后端分离架构，RESTful API 设计
- **Mock-First 开发模式**：通过策略模式将 OCR、AI 文本、AI 视觉、短信四个外部依赖抽象为可切换的 Provider，实现零外部依赖的完整本地开发和端到端测试

## 面试可重点讲的技术亮点

1. **异步流水线设计** — Celery 7 阶段任务状态机，每阶段更新进度百分比和 ETA，支持任务重试和优雅降级（Redis 不可用时同步执行）
2. **Agent 安全架构** — 不允许 AI 直接调用写操作，通过 Tool Registry 控制权限，所有 AI 触发的操作记录审计日志（含前后业务状态快照），Observer 模式在操作后重读状态以检测副作用
3. **LLM 输出校验** — 多层校验管线：禁止敏感字段泄漏检查（40+ 内部字段黑名单）、数据一致性验证（LLM 输出的热量数字必须与系统数据一致）、时间一致性验证（不能在已知是早上时说"深夜不适合"）、禁止伪造不存在的数据（空数据时不能说"摄入了 X kcal"）
4. **双模式食物分析** — Vision AI 判断分析模式：`dish_with_components`（混合菜品需拆解子成分，如麻辣烫）vs `component_sum`（独立食物汇总，如减脂餐盒），不同模式使用不同 Prompt 策略
5. **Mock-First + Provider 模式** — 开发和测试阶段无需任何外部 API Key，策略模式切换 Mock/真实服务，Redis 异常时自动降级
6. **融合策略（Fusion）** — OCR 文字识别（适合营养标签）和 AI 视觉识别（适合无文字食物）各有优势，系统融合两者结果并标注每项数据的来源和置信度

## 面试官可能追问的问题和回答方向

**Q1: 为什么不用现成的 OpenAI/Claude API，而是用百炼（阿里云）？**
A: 技术层面，百炼提供了 OpenAI 兼容接口，代码通过 Provider 模式抽象，切换模型只需要改配置。选择百炼是因为项目面向国内用户，百炼的合规性和访问稳定性更好。架构上所有 AI 调用都通过统一的 client 抽象层，切换模型（包括切换到 Claude/OpenAI）不需要改业务代码。

**Q2: AI 助手的安全性怎么保证？如果 LLM 说"我帮你删除了记录"怎么办？**
A: 三层防护：(1) Tool Registry 声明每个工具是否只读、是否需要确认、风险等级，写操作必须用户在前端点击确认；(2) 输出校验管线检查 LLM 回答中是否包含"已帮你""已经删除"等禁止表述，出现则触发重试或回退；(3) Action Audit 记录所有 AI 触发操作的 before/after 快照，可审计回溯。LLM 声称执行了操作但实际没有执行时，校验系统会拦截。

**Q3: RAG 为什么用关键词匹配而不是向量检索？**
A: MVP 阶段优先可用性。知识库文档量不大（当前 ~25 篇），关键词匹配足够覆盖。向量检索（pgvector）在架构上已预留升级空间 — `rag_service.py` 的接口是抽象的，升级只需要替换内部实现，不影响上层调用。这是刻意设计的：先验证 RAG 流程是否正确（检索→注入→约束生成），再优化检索质量。

**Q4: 7 阶段流水线会不会太长？用户等待体验怎么处理？**
A: 每个阶段通常几十毫秒到几秒，总耗时主要花在 AI 视觉识别和 AI 文本生成上（各 2-5 秒）。前端通过轮询 + 实时进度条 + 阶段事件列表展示进度，用户体验类似"正在分析中..."的加载动画。Celery 异步执行，用户上传后可以离开页面，前端会在任务完成后自动跳转到结果页。

**Q5: 草稿/确认机制的设计理由是什么？**
A: AI 对食物克数和营养成分的判断无法做到 100% 准确。与其给用户一个看起来精确但实际可能有偏差的结果，不如把 AI 分析定位为"初稿"，用户可以手动调整食物名称、重量、成分后再确认。确认后的数据才进入统计，确保统计数据的可靠性。这是 AI 产品中"人机协作"的设计理念。

**Q6: 如果 Celery Worker 挂了怎么办？**
A: `task_service.py` 中有 try/except 包裹，Celery 不可用时任务会同步在当前进程中执行，前端用户体验不受影响（只是分析变慢了一些）。Redis 不可用时也有类似的降级处理。系统设计原则是优雅降级而非完全不可用。

**Q7: 数据库为什么有 11 个模型？是不是过度设计了？**
A: 11 个模型涵盖用户、食物记录、分析任务、统计、AI 日志、助手记忆、知识库等不同领域。每个模型职责单一（如 `AILog` 只记录 AI 调用审计，`AssistantActionLog` 只记录操作审计），遵循关注点分离原则。模型之间通过 UUID 外键关联，支持级联删除保证数据一致性。对于 MVP 来说可能偏多，但考虑到 AI 审计和助手功能的设计需求，这些模型从第一天就是为了支持后续功能迭代的。

**Q8: 这个项目最让你自豪的技术决策是什么？**
A: Mock-First + 校验先行的设计理念。Mock 模式让项目在任何环境下都能跑起来和测试，降低了开发门槛。校验管线让 LLM 输出变得可预测和可信任，这在 AI 应用中是很容易被忽略但非常重要的一环。这两个决策让项目从"AI Demo"变成了一个真正可用的产品。
